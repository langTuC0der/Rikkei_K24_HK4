Kiến trúc Transformer & Cách LLM sinh văn bản

Hình 1.1 — Kiến trúc xử lý ngôn ngữ của LLM: từ tokenization → attention → output generation

Large Language Models (LLMs) là các mô hình AI được huấn luyện trên lượng dữ liệu khổng lồ để hiểu và sinh ngôn ngữ tự nhiên. Dưới đây là các khái niệm cốt lõi:

  • Tokenization — Chia văn bản thành các "token" (đơn vị nhỏ nhất). Ví dụ: "Xin chào" có thể được chia thành ["X", "in", " ch", "ào"]. Mỗi model có tokenizer riêng.
  • Attention Mechanism — Cơ chế cho phép model "chú ý" đến các phần liên quan trong input. Đây là trái tim của kiến trúc Transformer (Google, 2017).
  • Next-Token Prediction — LLM hoạt động bằng cách dự đoán token tiếp theo dựa trên tất cả tokens trước đó. Nó KHÔNG "hiểu" — nó "dự đoán" dựa trên pattern.
  • Temperature & Top-p — Tham số kiểm soát "sự sáng tạo":
    • temperature = 0: Output deterministic, luôn chọn token có xác suất cao nhất
    • temperature = 1: Output đa dạng, sáng tạo hơn
    • top-p (nucleus sampling): Chỉ xét các token có tổng xác suất đạt ngưỡng p
💡
Tại sao điều này quan trọng?

Hiểu cách LLM hoạt động giúp bạn viết prompt hiệu quả hơn. Khi biết model "dự đoán token tiếp theo", bạn sẽ hiểu tại sao instruction rõ ràng = output tốt hơn (vì nó giảm entropy — giảm số khả năng token tiếp theo).

Sự tiến hóa Context Window — Từ 4K đến 1M+ tokens

Context Window là "bộ nhớ làm việc" của LLM — tổng số tokens mà model có thể xử lý cùng lúc (bao gồm cả input và output). Đây là giới hạn quan trọng nhất bạn cần biết:

Context Window Evolution: 2022 │ GPT-3.5 │ 4,096 tokens (~3,000 words) 2023 │ GPT-4 │ 8,192 tokens (~6,000 words) │ GPT-4 Turbo │ 128,000 tokens (~96,000 words) 2024 │ Claude 3 │ 200,000 tokens (~150,000 words) │ Gemini 1.5 Pro │ 1,000,000 tokens (~750,000 words) 2025 │ GPT-5 │ 128,000+ tokens │ Claude Opus 4 │ 200,000 tokens 2026 │ Claude Fable 5 │ 200,000+ tokens │ GPT-5.5 │ 128,000+ tokens │ Gemini 3.5 Flash │ 1,000,000+ tokens ← LARGEST!

Reasoning Models vs. Standard Models

GPT-5 Pro và Gemini 3.5 Pro là "reasoning models" — chúng suy nghĩ sâu hơn (internal chain-of-thought) trước khi trả lời. Dùng cho bài toán logic phức tạp, math, code debugging. Standard models (GPT-5.5, Claude Opus 4.8) phù hợp cho task hàng ngày, nhanh hơn và rẻ hơn.

Ba trụ cột của AI Engineering

Hình 1.3 — Mối quan hệ 3 tầng: Prompt ⊂ Context ⊂ Harness Engineering
  • Prompt Engineering — Nghệ thuật viết instruction để LLM hiểu đúng ý bạn. "Viết câu hỏi thông minh"
  • Context Engineering — Khoa học quản lý thông tin đầu vào (context window) — quyết định LLM biết gì khi trả lời. "Cung cấp đúng thông tin"
  • Harness Engineering — Thiết kế hệ thống bao bọc LLM — orchestration, guardrails, pipelines, tool-use. "Xây dựng hệ thống điều phối"
Mối quan hệ phụ thuộc: Prompt ⊂ Context ⊂ Harness ┌─────────────────────────────────────────────────┐ │ HARNESS ENGINEERING │ │ ┌───────────────────────────────────────────┐ │ │ │ CONTEXT ENGINEERING │ │ │ │ ┌─────────────────────────────────────┐ │ │ │ │ │ PROMPT ENGINEERING │ │ │ │ │ │ "Viết câu hỏi thông minh" │ │ │ │ │ └─────────────────────────────────────┘ │ │ │ │ "Cung cấp đúng thông tin" │ │ │ └───────────────────────────────────────────┘ │ │ "Xây dựng hệ thống điều phối" │ └─────────────────────────────────────────────────┘
💡
Analogy dễ hiểu

Prompt Engineering giống viết email hay → Context Engineering giống chuẩn bị toàn bộ hồ sơ đính kèm → Harness Engineering giống xây cả hệ thống email + workflow + approval chain.

🔄 🆕 Paradigm Shift 2025-2026

Sự dịch chuyển: Từ Prompt Engineering → Context Engineering

Mô hình Karpathy: LLM là CPU, Context Window là RAM, Bạn là OS

Hình 1.4 — Mô hình Karpathy: Engineer (OS) quản lý Context Window (RAM) cho LLM (CPU) xử lý

Andrej Karpathy — cựu Director of AI tại Tesla, co-founder OpenAI — đã đưa ra mô hình mental model đỉnh cao để hiểu vai trò của mỗi thành phần:

Karpathy Mental Model (2025-2026): ┌──────────────────────────────────────────────────────┐ │ YOUR AI SYSTEM │ │ │ │ ┌──────────┐ │ │ │ YOU │ ← Hệ điều hành (OS) │ │ │ (Engineer)│ Quản lý memory, schedule tasks │ │ └────┬─────┘ │ │ │ loads context into │ │ ▼ │ │ ┌──────────────────────────────────────┐ │ │ │ CONTEXT WINDOW (RAM) │ │ │ │ ┌──────────┐ ┌──────────┐ ┌──────┐ │ │ │ │ │System │ │Retrieved │ │User │ │ │ │ │ │Prompt │ │Documents │ │Query │ │ │ │ │ └──────────┘ └──────────┘ └──────┘ │ │ │ │ ┌──────────┐ ┌──────────┐ │ │ │ │ │Tool │ │Chat │ │ │ │ │ │Schemas │ │History │ │ │ │ │ └──────────┘ └──────────┘ │ │ │ └──────────────────────────────────────┘ │ │ │ processes │ │ ▼ │ │ ┌──────────┐ │ │ │ LLM │ ← CPU (xử lý dựa trên context) │ │ │ (Model) │ │ │ └──────────┘ │ │ │ └──────────────────────────────────────────────────────┘
  • Bạn (OS) — Quyết định load gì vào RAM, khi nào load, khi nào xóa. Đây là kỹ năng Context Engineering.
  • Context Window (RAM) — Bộ nhớ làm việc có giới hạn. Chứa: system prompt, documents, tools, chat history, user query.
  • LLM (CPU) — Chỉ xử lý dựa trên những gì có trong RAM. Nếu info không có trong context → model không biết.
💡
Hệ quả quan trọng

Garbage In, Garbage Out → trở thành Context In, Quality Out. Model chỉ tốt khi context tốt. Vì vậy, việc quản lý context (chọn đúng thông tin, loại bỏ noise, sắp xếp ưu tiên) quan trọng hơn viết prompt hay.

Steerability: Điều khiển output của LLM

Steerability là khả năng điều khiển output của LLM theo hướng mong muốn. Prompt rõ ràng = steerability cao = output chất lượng.

So sánh Prompt 
❌ PROMPT MƠ HỒ (Low Steerability):
"Viết code"
→ Model không biết: ngôn ngữ nào? Task gì? Style nào?

✅ PROMPT RÕ RÀNG (High Steerability):
"Viết hàm Python sắp xếp danh sách số nguyên
 bằng thuật toán merge sort.
 Yêu cầu: có docstring, type hints, xử lý edge cases
 (empty list, single element).
 Output format: chỉ code, không giải thích."
→ Model biết chính xác: Python, merge sort, có constraints
🆕
2026 Update: Reasoning Models không cần "think step by step"

Với GPT-5 Pro và Gemini 3.5 Pro, việc thêm "hãy suy nghĩ từng bước" có thể KHÔNG CẦN THIẾT hoặc thậm chí CÓ HẠI — vì model đã tự reasoning bên trong. Thêm instruction CoT có thể gây "double-reasoning" và tăng latency. Tuy nhiên, với standard models (GPT-5.5, Claude Opus 4.8), CoT vẫn hiệu quả cho bài toán phức tạp.

Agentic Workflows — AI tự động hành động

Hình 1.6 — Agentic Workflow: AI agent tự lập kế hoạch, gọi tools (MCP), thực thi, và báo cáo kết quả

Năm 2026, AI không chỉ "trả lời câu hỏi" — mà tự hành động. Đây là xu hướng lớn nhất:

  • Background Agents — Tasks chạy hàng phút/giờ trong nền, webhook thông báo khi xong. Tools: Claude Cowork, Gemini Spark
  • Workspace Agents — Shared agents trong tổ chức, quản lý workflow độc lập (OpenAI Enterprise)
  • Coding Agents — AI tự viết, test, chạy, và sửa code autonomously
Agentic Workflow Architecture (2026): ┌──────────────┐ │ User Intent │ "Phân tích doanh thu Q1 2026" └──────┬───────┘ ▼ ┌──────────────┐ ┌────────────────────┐ │ Agent Router │────▶│ Background Agent │ │ (Orchestrator)│ │ (async, long tasks)│ └──────┬───────┘ └────────┬───────────┘ │ │ ▼ ▼ ┌──────────────┐ ┌────────────────────┐ │ Tool Calls │ │ Webhook │ │ via MCP │ │ Notification │ │ (database, │ │ "Task completed!" │ │ search, │ └────────────────────┘ │ compute) │ └──────────────┘
🔌
MCP (Model Context Protocol) — "USB-C for AI"

MCP là chuẩn open-source (Anthropic) cho phép AI agent kết nối với mọi tools & data sources qua 1 giao thức thống nhất. Thay vì viết custom integration cho từng tool, bạn chỉ cần build 1 MCP server → mọi MCP-compliant agent đều dùng được. Bạn sẽ học chi tiết ở Session 4 & 5.

❌ Với reasoning models, "think step by step" có thể KHÔNG CẦN hoặc CÓ HẠI vì model đã tự reasoning. Chỉ hiệu quả với standard models cho bài toán phức tạp.
