Lesson 01

Kiến trúc AI Agent & Chat Memory

1. AI Agent là gì?

Khác với mô hình ngôn ngữ (LLM) thông thường chỉ nhận Input và tạo ra Output (dạng Stateless), AI Agent là một hệ thống có khả năng tự động thực hiện các hành động phức tạp bằng cách lặp lại chu kỳ: Reasoning (Tư duy) -> Acting (Hành động) -> Observing (Quan sát).

So sánh trực quan
LLM thông thường: Giống như một nhà thông thái chỉ biết trả lời câu hỏi dựa trên kiến thức tĩnh có sẵn.
AI Agent: Giống như một nhân viên có máy tính, có quyền truy cập Internet, database và có thể tự thao tác thay bạn để xử lý công việc từ A-Z.

2. Mô hình ReAct (Reason + Act)

ReAct là framework kết hợp khả năng lập luận lý thuyết của LLM với khả năng tương tác thế giới ngoài thông qua các công cụ (Tools). Nếu không có vòng lặp ReAct, LLM sẽ cố gắng "đoán mò" và dẫn đến hiện tượng ảo tưởng (hallucination) khi gặp dữ liệu động.

Chu trình hoạt động của ReAct
Thought
LLM lập luận phương án
Action
Gọi API / Database
Observation
Nhận & đánh giá kết quả

3. Chat Memory — Trí nhớ hội thoại

Mặc định các cuộc gọi API đến LLM đều là Stateless. Để tạo trải nghiệm chat tự nhiên, ta cần gửi kèm toàn bộ lịch sử trò chuyện. Spring AI cung cấp interface ChatMemory để quản lý lịch sử này.

4. Quản lý Context Window (Cắt tỉa bộ nhớ)

Khi hội thoại quá dài, số lượng token sẽ vượt giới hạn (Context Window). Chúng ta cần các chiến lược quản lý:

  • Sliding Window: Chỉ giữ lại N tin nhắn gần nhất và loại bỏ các tin nhắn cũ hơn.
  • Token-based Truncation: Tự động loại bỏ tin nhắn cũ nhất khi tổng số token đạt ngưỡng 80%.
  • Summarization: Gọi một LLM phụ để tóm tắt các hội thoại cũ thành một đoạn văn ngắn cô đọng.

Luồng dữ liệu với Chat Memory

Dưới đây là sơ đồ tương tác tuần tự mô tả cách MessageChatMemoryAdvisor tự động hóa việc lưu trữ và đọc lịch sử trước khi gửi yêu cầu đến LLM.

LLM trả về câu trả lời. Advisor tự động ghi cả tin nhắn gửi đi và phản hồi vào ChatMemory trước khi trả về Client.

Lesson 02

Cơ chế Function Calling trong Spring AI

1. Function Calling là gì?

Function Calling (Gọi hàm) là cơ chế cho phép các mô hình ngôn ngữ lớn (như GPT-4, Gemini, Qwen) kết nối trực tiếp với các API bên ngoài hoặc chạy các đoạn mã lập trình nội bộ của bạn.

Sự thật quan trọng
LLM không trực tiếp chạy code Java trên server của bạn. Thay vào đó, nó nhận diện ý định của người dùng, tự động xuất ra một đối tượng cấu trúc dữ liệu JSON chứa tên hàm và tham số đầu vào cần thiết. Spring AI sẽ nhận JSON này, tìm và kích hoạt method Java tương ứng, rồi gửi trả lại kết quả thực thi về cho LLM.

2. Cơ chế định nghĩa Tool qua Functional Interface

Spring AI sử dụng các functional interface tiêu chuẩn của Java để định nghĩa công cụ, phổ biến nhất là java.util.function.Function<Request, Response>:

  • Tạo JSON Schema tự động: Spring AI tự động đọc cấu trúc của lớp Request (dùng Jackson và JSON Schema generator) để đặc tả cấu trúc tham số gửi đến LLM.
  • Mô tả tham số: Sử dụng annotation @JsonPropertyDescription là cực kỳ quan trọng để giải thích cho LLM biết biến đó có mục đích gì.
  • Ngữ cảnh kích hoạt: Sử dụng annotation @Description trên Bean để hướng dẫn AI khi nào thì nên sử dụng công cụ này.

Sơ đồ tuần tự của Function Calling

Mô phỏng luồng di chuyển dữ liệu 3 bên khi thực hiện yêu cầu gọi hàm:

1. User hỏi
"Đổi 100 USD sang VND"
2. LLM chọn Tool
Sinh tham số JSON: {amount:100}
3. Java Exec
Chạy hàm đổi tiền -> 2.54M VND
4. AI Answer
"100 USD bằng 2,540,000 VND"

Lesson 03

Đóng gói Spring Boot Service thành Tool

1. Khái niệm Tool trong Spring AI 1.1.5

Trong các phiên bản Spring AI gần đây (phiên bản 1.1.5 trở đi, giới thiệu từ bản 1.0.0.M1), một cơ chế đóng gói mới đã được giới thiệu giúp đơn giản hóa việc chuyển đổi các Service hiện có thành công cụ cho AI, đó là sử dụng annotation @Tool.

Trước đây, chúng ta phải khai báo thủ công từng Bean Functional Interface gây rườm rà và khó tái sử dụng code. Với @Tool, ta có thể trực tiếp đánh dấu một method trong `@Service` hoặc `@Component` để biến nó thành công cụ cho AI gọi.

2. Ưu điểm của Annotation @Tool

  • Dễ tích hợp: Không cần cấu hình lại các method dưới dạng Functional Interface. Một Service có thể chứa nhiều phương thức @Tool khác nhau.
  • Hỗ trợ tự động tạo Schema từ Javadoc: Spring AI tự động đọc Javadoc và các tên biến của phương thức để sinh ra bản đặc tả gửi đến LLM, giảm tải việc cấu hình thủ công.
  • Hỗ trợ @ToolContext: Cho phép truyền dữ liệu trạng thái (state) hoặc thông tin bảo mật (như JWT token, User ID) xuyên suốt cuộc gọi của Agent mà LLM không được phép nhìn thấy.
Mẹo viết Tool tối ưu
Khi sử dụng @Tool, hãy luôn cung cấp thuộc tính description thật rõ ràng hoặc viết Javadoc đầy đủ cho phương thức và các tham số. AI sẽ dựa vào mô tả này để quyết định có chọn gọi hàm này hay không.

Lesson 04

Dự án thực chiến: Hotel Booking Agent

1. Thiết kế Đa công cụ (Multi-Tool Agent)

Trong thực tế, một yêu cầu đặt phòng khách sạn của khách hàng không thể giải quyết bằng một công cụ đơn lẻ. AI Agent cần có khả năng tự động lập kế hoạch và kết hợp nhiều công cụ theo chuỗi để hoàn thành mục tiêu.

Ví dụ: Kiểm tra phòng trống -> Báo giá cho khách hàng -> Nhận xác nhận -> Thực hiện đặt chỗ.

2. Tầm quan trọng của sự kết hợp: Chat Memory + Tool Calling

Để Agent hoạt động chính xác trong cuộc đối thoại dài, nó cần sự phối hợp nhịp nhàng giữa:

  • Chat Memory: Nhớ các chi tiết người dùng đã cung cấp ở các câu trước (loại phòng, số đêm, ngày nhận phòng).
  • Tool Calling: Lấy dữ liệu thực tế tại thời điểm gọi hàm để đưa ra quyết định hoặc thực hiện ghi nhận giao dịch vào database.

3. An toàn giao dịch và tính toàn vẹn dữ liệu

Khi Agent gọi các hàm làm thay đổi trạng thái hệ thống (ghi/xóa database), chúng ta cần áp dụng các biện pháp bảo vệ:

  • Idempotency (Tính khả định): Đảm bảo việc gọi lại Tool đặt phòng không tạo ra các bản ghi trùng lặp (ví dụ: dùng ID giao dịch hoặc kiểm tra trạng thái trước khi lưu).
  • Xác nhận rõ ràng (Explicit Confirmation): Quy định rõ trong System Prompt rằng AI Agent bắt buộc phải hỏi lại và nhận được sự đồng ý rõ ràng từ khách hàng trước khi gọi Tool tạo đặt phòng (createBooking).
